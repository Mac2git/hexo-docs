Nginx

简介

Nginx（发音为 engine x）是一个开源的、高性能的HTTP和反向代理服务器，同时也提供了IMAP/POP3/SMTP邮件服务器。Nginx以其稳定性、高性能、丰富的功能集、简单的配置文件和低系统资源消耗而闻名。Nginx 的设计目标是提供一个高效的Web服务器，同时能够处理大量的并发连接和请求，

Nginx的架构基于事件驱动和异步非阻塞I/O，相比于传统的 Web 服务器，如Apache，Nginx 的内存占用更少，启动速度更快，在高并发场景下表现优异。

Nginx的功能非常丰富，包括静态文件服务、反向代理、负载均衡、HTTP/2支持、WebSocket代理、SSL/TLS加密、URL重写、访问控制、HTTP缓存等。同时，Nginx还支持大量的第三方模块，这些模块可以扩展Nginx的功能以满足特定的需求。

Nginx的核心功能

  • HTTP Web 服务器：基础功能，用于接收和响应 HTTP 请求，向客户端提供网页等静态资源。
  • 反向代理：接收客户端请求后，将请求转发给后端服务器，再将后端服务器的响应返回给客户端，可隐藏后端服务器真实地址，提升安全性。
  • 内容缓存：将后端服务器返回的常用内容缓存起来，后续客户端请求相同内容时，直接从缓存中读取，减少后端服务器压力，提高响应速度。
  • 负载均衡：将大量客户端请求均匀分配到多个后端服务器，避免单个服务器过载，保证服务稳定运行。
  • TCP/UDP 代理服务器：可对 TCP 和 UDP 协议的请求进行代理转发，适用于多种非 HTTP 协议的服务场景。
  • 邮件代理服务器：支持对邮件相关协议（如 SMTP、POP3、IMAP）的请求进行代理处理。

Nginx正则表达式解析

Nginx 是一款高性能的开源 Web 服务器，正则表达式在其配置中被广泛用于 URL 匹配、重定向和访问控制等操作。以下是 Nginx 正则表达式的基本用法和常见场景。

基本语法

Nginx 中的正则表达式通常以 ~ 或 ~* 开头：

  • ~ 表示区分大小写匹配。
  • ~* 表示不区分大小写匹配。
  • !~ 和 !~* 分别表示不匹配的区分大小写和不区分大小写。

常见符号包括：

  • ^ 表示以某内容开头。
  • $ 表示以某内容结尾。
  • .* 表示匹配任意字符。
  • ** 用于转义特殊字符。

常见用法

  • 精确匹配：使用 = 标识符。
  • 前缀匹配：使用 ^~ 标识符。
  • 正则表达式匹配：使用 ~ 或 *~**。
  • 反向引用：通过括号捕获匹配内容并在后续使用 $1、*$2* 等引用。

匹配优先级

Nginx 的 location 匹配遵循以下顺序：

  1. 精确匹配 (=)。
  2. 前缀匹配 (^~)。
  3. 正则表达式匹配（按配置顺序）。
  4. 通用匹配（*/*）。

Nginx负载均衡介绍

负载均衡就是当我们的服务器接收到的请求太多，压力太大，此时可能导致服务器性能下降或不可用，为了解决这个问题，我们可以搭建多台服务器，将请求分配到多个服务器上进行处理，从而保证服务的可用性，即使有服务器挂掉，服务也能正常运行，通过负载均衡，我们可以动态的增加或减少服务器的数量，增强网站的可扩展性，这样也利于我们更新服务。

而 Nginx 作为一款高性能的 Web 服务器和反向代理服务器，可以通过其负载均衡功能，将接收到的客户端请求，根据配置的负载均衡策略（如轮询、最少连接、IP哈希等）转发给多个后端服务器，从而实现负载均衡。

Nginx有5种负载均衡方式：

  1. 轮询（默认）：是nginx负载默认的方式，它的本意是将海量请求按照时间的顺序分配到不同的服务器上；如果某个服务器down掉，Nginx可以将其自动踢出集群。

  2. 权重：指定每个服务器的权重比例，由于有的服务器性能比较好，所以通过设置不同服务器间权重的比例可以发挥服务器的最大性能。权重越高被分配的几率越大，默认是1

  3. ip_hash：根据ip经过hash后的结果进行分配，这样每个ip只能固定访问一个服务，可以解决session的问题

  4. 最小连接：将请求分配到连接最少的服务上

  5. fair（第三方）：将后端服务器的响应时间来分配请求，响应时间短的优先分配。可以更好保证服务器的性能的问题！

案例四（动静分离）

Nginx 动静分离简单来说就是把动态跟静态请求分开，不能理解成只是单纯的把动态页面和静态页面物理分离。严格意义上说应该是动态请求跟静态请求分开，可以理解成使用 Nginx

处理静态页面，Tomcat 处理动态页面。动静分离从目前实现角度来讲大致分为两种，一种是纯粹把静态文件独立成单独的域名，放在独立的服务器上，也是目前主流推崇的方案；另外一种方法就是动态跟静态文件混合在一起发布，通过 nginx 来分开。

通过 location 指定不同的后缀名实现不同的请求转发。通过 expires 参数设置，可以使浏览器缓存过期时间，减少与服务器之前的请求和流量。具体 Expires 定义：是给一个资源设定一个过期时间，也就是说无需去服务端验证，直接通过浏览器自身确认是否过期即可，所以不会产生额外的流量。此种方法非常适合不经常变动的资源。（如果经常更新的文件，不建议使用 Expires 来缓存），我这里设置 3d，表示在这 3 天之内访问这个 URL，发送一个请求，比对服务器该文件最后更新时间没有变化，则不会从服务器抓取，返回状态码 304，如果有修改，则直接从服务器重新下载，返回状态码 200。

案例五（搭建高可用Nginx集群）

我们为了实现高可用，很明显一台nginx是不够的，所以我们要使用keepalived搭建nginx主从

keepalived是什么？

Keepalived 是一个用 C 语言编写的、基于 VRRP 协议 实现的，用于提供高可用性和负载均衡功能的开源软件。它的主要目标是简单、健壮，并且能有效防止单点故障。

核心功能与工作原理

Keepalived 的核心可以概括为两大功能：

1. 高可用性 - 核心功能

这是 Keepalived 最常用、最主要的功能。它用于确保网络服务的连续性和可靠性。

工作原理：

  • VRRP 协议： Keepalived 通过 VRRP（Virtual Router Redundancy Protocol，虚拟路由冗余协议）来实现高可用。VRRP 将多台物理服务器（通常是一个主节点和一个或多个备用节点）组合成一个虚拟的服务器（或称为虚拟路由器）。
  • 虚拟 IP： 这个虚拟服务器拥有一个虚拟 IP 地址。对外提供服务的正是这个 VIP，而不是任何一台真实服务器的 IP。
  • 角色选举：
    • 主节点： 在多台服务器中，通过优先级竞选出一台作为 Master（主节点）。Master 服务器会持有这个 VIP，并实际承担所有的用户请求和流量。
    • 备用节点： 其他服务器作为 Backup（备用节点），它们处于监听状态。
  • 心跳检测与故障转移：
    • 所有节点之间通过多播心跳消息来相互通信，检查对方是否存活。
    • 当 Backup 节点在一定时间内收不到 Master 节点的心跳消息时，它们会认为 Master 宕机了。
    • 此时，优先级最高的 Backup 节点会接管 VIP，将自己升级为新的 Master，开始提供服务。
    • 这个过程对用户来说是透明的，用户感知到的只是始终通过同一个 VIP 访问服务，并不知道后端是哪台服务器在响应。

简单比喻：
就像是一个团队的备胎机制。

  • VIP 是团队对外的统一热线电话。
  • Master 是当前接听热线的主力客服。
  • Backup 是旁边待命的替补客服，时刻听着主力客服的动静。
  • 一旦主力客服突然倒下（宕机），替补客服会立刻拿起电话（接管VIP），保证热线永远畅通。
2. 负载均衡 - 辅助功能

Keepalived 自身也具备第4层（传输层，TCP/UDP）的负载均衡能力，它通过集成并管理 Linux Virtual Server 的 IPVS 内核模块来实现。

  • IPVS： 这是一个工作在内核层面的、高性能的负载均衡器。
  • Keepalived 的角色： Keepalived 充当了 IPVS 的配置和管理工具。你可以通过在 Keepalived 的配置文件中定义virtual_server段，来告诉 IPVS 如何将到达 VIP 的流量分发到后端的多个真实服务器上。

需要注意的是： 在实际生产中，更常见的做法是使用 Keepalived 实现高可用 + 其他专用负载均衡器（如 Nginx, HAProxy） 的组合。即：

  • Keepalived 负责 VIP 的漂移，保证 Nginx/HAProxy 本身的高可用。
  • Nginx/HAProxy 负责更强大的第7层（应用层）负载均衡、健康检查、SSL termination 等复杂功能。

主要特点

  1. 轻量级： 资源占用少，性能高。
  2. 健壮性： 代码简洁，故障转移速度快（通常在几秒内完成）。
  3. 配置简单： 主要通过单个配置文件进行管理，易于理解和部署。
  4. 开源免费： 遵循 GPL 协议。

典型应用场景

  1. Web 服务器高可用： 为 Nginx、Apache 等 Web 服务器提供故障转移。
  2. 数据库高可用： 为 MySQL、Redis 等数据库主从架构提供 VIP 漂移。
  3. 负载均衡器高可用： 如上文所述，为 Nginx、HAProxy、LVS 自身提供高可用，形成“双机热备”架构。这是最经典的用法。
  4. 任何需要避免单点故障的 TCP 服务。
